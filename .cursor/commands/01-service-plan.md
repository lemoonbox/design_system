# /service-plan

## 목적
사용자와 함께 서비스 기획을 진행하고 `service-plan.md`를 작성한다.
이 문서는 이후 모든 단계의 기준이 된다.

## 진행 방식
한 번에 하나씩 대화형으로 질문한다. 리스트로 한꺼번에 묻지 않는다.
사용자의 답변을 바탕으로 다음 질문을 이어간다.

---

## 인터뷰 흐름

### Step 1. 서비스 목적
- 이 서비스는 어떤 문제를 해결하는가?
- 현재 그 문제를 어떻게 해결하고 있는가? (현황)
- 이 서비스가 없으면 어떤 불편이 생기는가?

### Step 2. 사용자
- 주요 사용자는 누구인가? (역할, 직군, 상황)
- 사용자가 이 서비스를 언제, 어디서 사용하는가?
- 사용자가 가장 자주 하는 행동 1~3가지는?

### Step 3. 핵심 기능
- 서비스에서 반드시 있어야 하는 기능은?
- 있으면 좋지만 없어도 되는 기능은?
- 명시적으로 제외하고 싶은 것은?

### Step 4. 도메인 데이터
Step 3에서 나온 각 기능을 하나씩 짚으며 데이터를 구체화한다.

- **기능별 데이터 항목**: 이 기능이 다루는 구체적 데이터 필드는 무엇인가? (입력·출력·참조 각각)
  - 예: "구급 일지"라면 → 활력징후(혈압·맥박·SpO2·체온), 의식수준(AVPU), 주호소, 처치내역, 시간 기록 등
- **현행 워크플로 데이터 소스**: 현재 업무에서 참고하는 양식·문서·화면·종이 서류는? (기존 데이터 원천)
  - 예: 구급활동일지 양식, 병원 연락 시 확인하는 항목 등
- **의사결정 기준 정보**: 사용자가 판단·비교할 때 보는 기준은 무엇인가?
  - 예: 병원 선택 시 → 거리, 전문과, 여유 병상, 수술 가능 여부, 과거 수용 이력 등

> 이 단계에서 도출된 데이터는 이후 IA·화면 스펙에서 화면 구성의 재료가 된다. 가능한 한 구체적으로 끌어낸다.

### Step 5. 비즈니스 목표
- 이 서비스로 달성하고 싶은 목표는? (효율화 / 매출 / 사용자 경험 개선 등)
- 성공을 어떻게 측정할 것인가?

### Step 6. 제약 조건
- 플랫폼 제약이 있는가? (모바일 전용 / 웹 / 데스크탑 앱 등)
- 기존 시스템과 연동이 필요한가?
- 디자인적으로 따라야 하는 브랜드 가이드나 규정이 있는가?

### Step 7. 확인 및 정리
수집된 내용을 요약하여 보여주고 수정 사항을 확인한다.

## 출력: service-plan.md

```markdown
# Service Plan — {서비스명}

## 서비스 개요
- 목적: (한 문장)
- 해결하는 문제: (현황 포함)

## 사용자
- 주요 사용자: (역할/상황)
- 사용 맥락: (언제, 어디서)
- 핵심 행동: (1~3가지)

## 기능 범위
### 필수 기능
- ...

### 선택 기능
- ...

### 제외
- ...

## 도메인 데이터

### 기능별 데이터 항목
| 기능 | 입력 데이터 | 출력·표시 데이터 | 참조 데이터 |
|------|-----------|----------------|-----------|
| {기능1} | ... | ... | ... |

### 현행 데이터 소스
- (기존 양식·문서·시스템 등)

### 의사결정 기준
| 판단 상황 | 비교 기준 | 우선순위 |
|----------|----------|---------|
| {상황1} | ... | ... |

## 비즈니스 목표
- 목표: ...
- 성공 지표: ...

## 제약 조건
- 플랫폼: ...
- 연동 시스템: (있다면)
- 브랜드/규정: (있다면)
```

저장 위치: `projects/{nn}.{project-name}/context/service-plan.md`
