# /ia-flow

## 목적
`service-plan.md`를 기반으로 IA(정보 구조)와 서비스 플로우를 작성한다.
이후 `screen-spec` 작성 시 필수 데이터와 이동 조건의 기준이 된다.

## 실행 전 읽기
- `projects/{nn}.{project-name}/context/service-plan.md` — **`## 도메인 데이터`** 섹션을 반드시 확인. 기능별 입력·출력·참조 데이터가 화면 구성의 재료.

## 진행 방식
사용자와 함께 구조를 논의하며 작성한다.
확정 전 초안을 보여주고 수정 의견을 받는다.

---

## Step 1. 화면 목록 도출
service-plan의 핵심 기능 + **도메인 데이터**를 기준으로 필요한 화면을 나열한다.
- 각 기능을 수행하려면 어떤 화면이 필요한가?
- 중복되거나 합칠 수 있는 화면은 없는가?
- 사용자가 보게 되는 첫 화면은 무엇인가?
- **각 화면이 표시해야 하는 핵심 정보를 함께 정리한다** (service-plan `도메인 데이터`에서 도출)

## Step 2. IA 구조 결정
- 화면을 계층으로 분류한다 (Global / Local / Modal)
- 네비게이션 패턴을 결정한다 (하단 탭 / 사이드바 / 단계형 / 드로어)
- 사용자와 함께 확인 후 확정한다

## Step 3. 서비스 플로우 작성
핵심 사용자 행동 경로를 텍스트 흐름으로 표현한다.
- 정상 플로우 (happy path)
- 오류/예외 플로우
- 화면 간 이동 조건 (예: 저장 완료 시 → 목록 이동)
- **각 플로우 단계에서 사용자가 확인하는 정보를 `[정보: ...]` 주석으로 명시한다**

## Step 4. 대표 화면 와이어프레임 검증

> 이 단계의 목적: IA·플로우가 확정되기 전에, 주요 화면이 **실제로 충분한 정보와 구성 요소를 담는지** 시각적으로 확인한다.

**Global·Local 화면**(모달 제외)마다 마크다운 텍스트 와이어프레임을 그린다.

작성 규칙:
- 박스(`┌─┐`)와 텍스트로 화면 구조를 표현
- 각 영역에 **표시될 데이터 필드명**을 기입
- 카드가 반복되면 1장만 그리고 `× N` 표기
- 와이어프레임 아래에 **정보 밀도 자기 점검**을 기술:
  - 카드 내 데이터 필드가 3개 미만이면 → service-plan 도메인 데이터에서 보조 정보 추가 검토
  - 화면 주요 영역의 50% 이상이 빈 공간이면 → 맥락 정보·보조 표시 추가 검토

**사용자 확인**: 와이어프레임을 보여주고 다음을 질문한다:
1. 이 화면에 빠진 정보가 있는가?
2. 불필요한 정보가 있는가?
3. 정보의 우선순위(배치)가 적절한가?

**service-plan 피드백**: 확인 과정에서 service-plan에 없던 데이터 항목이나 기능 변경이 필요하면:
1. 변경 필요 사항을 사용자에게 명시적으로 제시한다
2. **사용자 허락을 받은 후에만** service-plan.md를 수정한다
3. 수정 내용을 ia.md에도 반영한다

## 출력: ia.md

```markdown
# IA & Service Flow — {서비스명}

## 화면 목록
| ID | 화면명 | 레벨 | 설명 | 핵심 정보 |
|----|--------|------|------|----------|
| G1 | 목록 | Global | 전체 항목 탐색 | {이 화면에서 표시하는 주요 데이터 항목} |
| L1 | 상세 | Local | 단일 항목 상세 보기 | {주요 데이터 항목} |
| L2 | 편집 | Local | 항목 데이터 입력/수정 | {입력·표시 데이터 항목} |
| M1 | 확인 다이얼로그 | Modal | 삭제/제출 확인 | {확인 대상 요약 정보} |

## IA 구조
{텍스트 트리}
- [G1] 목록
  - [L1] 상세
    - [L2] 편집
    - [M1] 확인 다이얼로그

## 네비게이션 패턴
- 타입: (하단 탭 / 사이드바 / 단계형 등)
- 뒤로가기: (스택 팝 / 닫기 버튼 / 없음)
- 진입 불가 조건: (있다면. 예: 권한 없을 시 G1으로 리다이렉트)

## 서비스 플로우

### 플로우 1. {핵심 행동명}
```
[G1 목록]
  → 항목 탭  [정보: 항목 제목, 상태 배지, 최종 수정 시각]
  → [L1 상세]  [정보: 전체 필드, 첨부, 이력]
  → '편집' 버튼 탭
  → [L2 편집]  [정보: 편집 가능 필드, 현재 값]
  → 저장 완료
  → [L1 상세] (업데이트된 데이터)
```

### 플로우 2. {예외/오류 케이스}
```
[L2 편집]
  → 필수 항목 미입력 상태로 저장 시도
  → 인라인 오류 표시 (화면 유지)
  → 수정 후 재저장
  → [L1 상세]
```

## 화면 간 이동 조건 요약
| 출발 | 도착 | 조건 | 전달 데이터 |
|------|------|------|-----------|
| G1 | L1 | 항목 탭 | 항목 ID |
| L1 | L2 | 편집 버튼 탭 | 항목 전체 데이터 |
| L2 | L1 | 저장 성공 | 업데이트된 항목 |
| L2 | L2 | 저장 실패 (유지) | — |

## 대표 화면 와이어프레임

### {ID} — {화면명}
```
┌──────────────────────────────┐
│ 헤더: {제목}    {상태 배지}  │
├──────────────────────────────┤
│ ┌──────────────────────────┐ │
│ │ {데이터 필드1}           │ │
│ │ {데이터 필드2} {필드3}   │ │
│ │ {보조 정보}              │ │
│ └──────────────────────────┘ │
│         × N                  │
├──────────────────────────────┤
│ [ 주요 CTA ]                 │
└──────────────────────────────┘
```
**정보 밀도 점검**: 카드 필드 N개, 빈 영역 비율 약 N%
```

저장 위치: `projects/{nn}.{project-name}/context/ia.md`
