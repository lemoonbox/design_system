# /ia-flow

## 목적
`service-plan.md`를 기반으로 IA(정보 구조)와 서비스 플로우를 작성한다.
이후 `screen-spec` 작성 시 필수 데이터와 이동 조건의 기준이 된다.

## 실행 전 읽기
- `projects/{nn}.{project-name}/context/service-plan.md`

## 진행 방식
사용자와 함께 구조를 논의하며 작성한다.
확정 전 초안을 보여주고 수정 의견을 받는다.

---

## Step 1. 화면 목록 도출
service-plan의 핵심 기능을 기준으로 필요한 화면을 나열한다.
- 각 기능을 수행하려면 어떤 화면이 필요한가?
- 중복되거나 합칠 수 있는 화면은 없는가?
- 사용자가 보게 되는 첫 화면은 무엇인가?

## Step 2. IA 구조 결정
- 화면을 계층으로 분류한다 (Global / Local / Modal)
- 네비게이션 패턴을 결정한다 (하단 탭 / 사이드바 / 단계형 / 드로어)
- 사용자와 함께 확인 후 확정한다

## Step 3. 서비스 플로우 작성
핵심 사용자 행동 경로를 텍스트 흐름으로 표현한다.
- 정상 플로우 (happy path)
- 오류/예외 플로우
- 화면 간 이동 조건 (예: 저장 완료 시 → 목록 이동)

---

## 출력: ia.md

```markdown
# IA & Service Flow — {서비스명}

## 화면 목록
| ID | 화면명 | 레벨 | 설명 |
|----|--------|------|------|
| G1 | 목록 | Global | 전체 항목 탐색 |
| L1 | 상세 | Local | 단일 항목 상세 보기 |
| L2 | 편집 | Local | 항목 데이터 입력/수정 |
| M1 | 확인 다이얼로그 | Modal | 삭제/제출 확인 |

## IA 구조
{텍스트 트리}
- [G1] 목록
  - [L1] 상세
    - [L2] 편집
    - [M1] 확인 다이얼로그

## 네비게이션 패턴
- 타입: (하단 탭 / 사이드바 / 단계형 등)
- 뒤로가기: (스택 팝 / 닫기 버튼 / 없음)
- 진입 불가 조건: (있다면. 예: 권한 없을 시 G1으로 리다이렉트)

## 서비스 플로우

### 플로우 1. {핵심 행동명}
```
[G1 목록]
  → 항목 탭
  → [L1 상세]
  → '편집' 버튼 탭
  → [L2 편집]
  → 저장 완료
  → [L1 상세] (업데이트된 데이터)
```

### 플로우 2. {예외/오류 케이스}
```
[L2 편집]
  → 필수 항목 미입력 상태로 저장 시도
  → 인라인 오류 표시 (화면 유지)
  → 수정 후 재저장
  → [L1 상세]
```

## 화면 간 이동 조건 요약
| 출발 | 도착 | 조건 |
|------|------|------|
| G1 | L1 | 항목 탭 |
| L1 | L2 | 편집 버튼 탭 |
| L2 | L1 | 저장 성공 |
| L2 | L2 | 저장 실패 (유지) |
```

저장 위치: `projects/{nn}.{project-name}/context/ia.md`
