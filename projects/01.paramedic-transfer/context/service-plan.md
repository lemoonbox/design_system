# Service Plan — 구급 이송 조율 모바일 앱

## 서비스 개요

- **목적**: 앰뷸런스 구급대원이 현장에서 환자를 평가하고, 증상에 맞는 병원으로 이송을 조율할 수 있도록 돕는다.
- **해결하는 문제**: 병원 선정·이송 조율을 **전화**로 하고, **센터 복귀 후 PC**에 기록하는 현재 방식에서는 **기억 소실**이 크고, **업무가 매우 바쁜 상황에서 수기 메모 부담**이 크다.

## 사용자

- **주요 사용자**: 앰뷸런스 구급대원
- **사용 맥락**: 환자를 만난 뒤, **현장 도착 후 처치를 시작하는 시점부터** 사용. 처치 시작과 함께 **기록이 자동으로 시작**된다고 가정한다.
- **핵심 행동** (앱 관점):
  1. 음성 기반으로 구급 일지가 자동 기록되도록 말한다 (STT·NER). **구급 일지 화면에서는 STT·진행 표시를 소형 UI**로 둔다.
  2. **pre-KTAS·증상 보완은 별도 화면(L1)**에서 진행하며, **pre-KTAS와 병렬**인 **P-class**가 있다. **Class I~V 서브클래스 멀티 선택**; **Class III**(ICU·NICU·IR)는 **병원 추천 맥락에 함께 반영**; **Class II `delivery`** 는 **기본 제안 체크**. **주증상만** 있어도 진행 가능. **비동기**로 다룬다.
  3. **구급 일지에서 병원 추천(이송) 화면으로 바로 이동**할 수 있고, 병원은 **추천 목록에서 고르거나 직접 검색**해 선택한 뒤 일괄 신청한다. **pre-KTAS를 사용자가 “확정”하기 전에도** 일괄 이송 신청은 **허용**한다.
  4. 병원 승인 상태를 확인하고(**거절 시 사유** 확인), 승인된 병원 중 한 곳을 골라 이송을 시작한 뒤 **완료 전용 화면**으로 마무리한다.

## 기능 범위

### 필수 기능

- 구급대원 음성 입력 시 **STT 및 NER**로 구급 일지 **자동 기록**. **MVP 맵핑·추출 우선 영역**은 **`ems-log-field-hierarchy`의 환자증상·환자평가·응급처치** 중심으로 둔다(다른 키는 단계적 확장). 일지 항목은 **수동 수정 가능**. **구급 일지(G1) 화면**에서 STT 관련 표시는 **작게(보조 영역)** 유지
- **pre-KTAS·증상 평가**는 **일지와 분리된 전용 화면(L1)**에서 수행하고, **비동기 처리**(대기·완료·재시도, 처리 중 타 화면 이동)를 전제로 한다
- **P-class 환자 평가**는 **pre-KTAS와 병렬**인 **별도 파이프라인**이며, **서브클래스 멀티 선택**으로 기록된다. **주증상 선택**만으로도(자동 제안 등) 진행 가능하고, **pre-KTAS 확정을 기다리지 않는다**
- **이송 가능 병원 추천** 및 **직접 검색**으로 병원을 찾아 **동일한 선택 풀**에 담은 뒤 일괄 신청한다. **구급 일지에서 병원 단계로 바로 진입** 가능. **pre-KTAS “확정” 전에도** 일괄 이송 신청 **가능**(막지 않음; 추천 맥락은 일지·**주증상·P-class(선택 서브클래스)**·(있으면) pre-KTAS 순으로 보강)
- 구급대원이 **복수 병원을 선택**한 뒤 **한 번에 이송 신청**(일괄 신청). **동시 신청 병원 수 상한은 두지 않음**(추천·선택 UI는 스크롤 등으로 모두 수용)
- 신청한 **병원 각각**이 이송을 **승인 또는 거절**하며, 앱에는 **병원별 승인 상태**가 표시된다. **거절 시 사유**는 **표시**한다(데이터 제공 시)
- **승인을 받은 병원 중 한 곳만** 골라 **이송 시작(출발)** 할 수 있다 — **승인 전에는 출발 선택 불가**. **이송 시작(M1) 확정 후**에는 **완료 전용 화면(L4)** 으로 이동한다

### 선택 기능

- 없음 (위 항목을 이번 앱 디자인 범위의 필수로 둠)

### 제외

- **병원용 웹** (별도 제공 가정, 이번 구현·디자인 범위에서 제외)
- **센터 PC 기록 연동**, **결제**, **환자·가족용 기능**
- **기존 시스템 연동**은 앱 **화면·카피에서 언급하지 않음**
- **전화 통화와 동시에 앱을 사용하는 시나리오**는 이번 디자인에서 고려하지 않음

## 도메인 데이터

### 구급일지 Key → Value → Subvalue 계층

- 앱의 **구급 일지(G1)·STT/NER 엔티티 매핑·스펙** 시 참고하는 **3단 필드 경로**는 `context/ems-log-field-hierarchy.md`에 둔다.
- 기계 처리용 **탭 구분 원본**은 같은 폴더의 `ems_log_source.tsv` (생성 스크립트: `build_ems_log_md.py`).
- 제공 목록의 **완전 중복 행**은 TSV에서 제거·병합했다. 원문과 **글자(예: 오탈자)·빈 칸**이 다르면 별도 행으로 남을 수 있다. **오탈자 정정**: `환자증상 > 기타 > 김각이상` → **`감각이상`**(TSV·계층 문서 반영).

### P-class 체계 (멀티 선택 서브클래스)

- **정의**: P-class는 **5개 Class** 아래 **서브클래스(항목)** 를 **복수 선택**하는 구조다. **서로 다른 Class**에서 동시에 여러 항목을 고를 수 있다.
- **원본 계층**(기획 표 기준):

| Class | 명칭 | 서브클래스 (멀티 선택) |
|-------|------|------------------------|
| **I** | Resuscitation | ACLS/PALS, Intubation, Central catheterization, Massive transfusion |
| **II** | Emergency intervention | ATLS, PTCA, Thrombolysis, Embolization, Brain op, VDA, CRRT, preterm, delivery, burn, extremity(사지손상) |
| **III** | Special admission (원본 표기 **고려X**) | ICU, NICU, IR — **앱 정책**: **병원 추천·매칭 맥락에 함께 반영**(선택 시 추천 엔진이 동일하게 고려). UI는 스펙에서 상세화 |
| **IV** | Special evaluation | eye, ps, dent, ent, ob(산과), psy |
| **V** | Conservative management | pKTAS 1,2 · pKTAS 3 · pKTAS 4,5 — **pre-KTAS와 대응**하는 보존적 관리 구간 |

- **디자인 메모**: Class II **`delivery`** — **기본 제안 시 체크 ON**(사용자가 해제 가능). 시각 강조는 `visual-direction`·스펙에서 보조 정의.

### 현행 데이터 소스

- **구급활동일지**(종이·전자): 활력징후, 의식수준, 주호소, 처치·투약, 시간대별 기록 등 — 앱의 일지·NER 항목 설계 시 참고하는 **업무상 관습적 항목**. **코드화된 key–value–subvalue**는 `ems-log-field-hierarchy.md`·`ems_log_source.tsv`를 따른다.
- **현장·이송 조율 시 관행**: 병원 연락 시 확인하던 **수용 가능 여부·전문·거리** 등 — 추천 카드·승인 화면에 **표시할 정보의 우선순위** 논의용 (외부 DB 연동은 디자인 범위에서 명시하지 않음).
- **pre-KTAS·증상 분류**: 교육·지침에 따른 **등급 판단 요소** — L1의 pre-KTAS 영역 필드·검증·상태 문구 참고.
- **P-class**: 위 표가 **마스터**에 가깝다. **Class III**는 **추천에 반영**; **delivery**는 **기본 제안 체크**. 멀티 선택 UI는 **screen-spec**·`visual-direction`에서 구체화.

### 의사결정 기준

| 판단 상황 | 비교·확인 기준 | 우선순위(기본 가정) |
|----------|----------------|---------------------|
| 추천·검색 병원 검토(L2) | **추천 탭**: 적합 요약·거리·사유 / **검색**: 키워드·결과 비교 | 추천·검색 **동일 규칙으로 복수 선택** 후 일괄 신청. **환자 안전·필요 처치** 우선 |
| 이송 신청 대상 선택(L2) | 추천·검색 결과에서 **복수 체크**, **최소 1곳 이상**, **상한 없음** | **선택한 모든 병원**에 동시 신청; L3에서 병원별 응답. **스크롤·탭 전환**으로 다건 UI 수용 |
| G1에서 L2 직접 진입 | 일지·**주증상·P-class**(가용 시)만으로도 추천 요청 **허용**; pre-KTAS **미확정**이어도 **일괄 신청 허용** | L1은 **선택적** 보완 |
| 일괄 이송 신청(L2) | **pre-KTAS “확정” 불필요** — 신청 자체를 **차단하지 않음** | 추천·신청 페이로드에 **가용 맥락**만 실음 |
| P-class vs pre-KTAS | **병렬**; P-class는 **서브클래스 멀티 선택** + **주증상** 맥락; Class V는 **pKTAS 구간**과 **대응** | pre-KTAS 확정을 P-class **완료 조건으로 두지 않음** |
| P-class 멀티 선택(L1) | Class I~V **복수 선택**; **delivery** 제안 시 **기본 체크** | **상한 없음**(스크롤·접기). **Class III** 선택값은 **L2 추천 맥락에 포함** |
| 승인·거절 표시(L3) | **승인**만 출발 후보; **거절** 행에 **사유 표시**(데이터 있을 때) | 갱신 시각·상태 |
| 승인 병원 중 최종 1곳·출발(L3→M1) | **승인** 1곳 선택 → **M1** 확인 | 이송 시작 요청 |
| 이송 완료(L4) | M1 **확정** 후 자동 이동 | **완료 전용** 요약 화면만 사용(동일 화면에 머무르지 않음) |
| pre-KTAS·증상 확정(L1·비동기) | 일지 근거 vs 현장 관찰 보완; **처리 중 G1·L2 이동 허용** | **임상적 보수적 판단**; 확정은 **신청 잠금 조건이 아님** |

- **플랫폼**: **iOS·Android** 모바일 앱
- **오프라인·네트워크 불안정**: UI·시나리오에서 **고려** (동기화·재시도·상태 표시 등 화면 수준)
